# 北邮毕业论文 Markdown 写作规范（RSS 项目版）

> 适用于 `thesis.md -> thesis.docx` 的导出链路。
> 目标不是追求通用 Markdown，而是约束出一份**可稳定检查、可稳定导出、可稳定排版**的论文源文件。

## 1. 文档骨架

### 1.1 顶层结构

建议按以下顺序组织：

1. 中文题目：`# ...`
2. 中文摘要：`## 摘要`
3. 英文题目：`# ...`
4. 英文摘要：`## ABSTRACT`
5. 目录占位：`## 目录`
6. 正文章节：`## 第一章 ...` 起
7. 参考文献：`## 参考文献`
8. 致谢：`## 致谢`
9. 附录：`## 附录`

### 1.2 一级标题范围

一级标题（Markdown 中的 `##`）建议只用于：

- 摘要
- ABSTRACT
- 目录
- 第 X 章 ...
- 参考文献
- 致谢
- 附录

## 2. 标题层级与编号

### 2.1 层级映射

- `#`：中英文题目
- `##`：一级标题 / 章标题
- `###`：二级标题
- `####`：三级标题

### 2.2 标题编号规则

- 一级章标题使用“第一章 / 第二章 / ...”形式。
- 二级标题使用 `X.Y 标题` 形式，例如：`### 3.2 RSS 源管理`
- 三级标题使用 `X.Y.Z 标题` 形式，例如：`#### 3.2.1 RSS 源注册与配置`
- 不允许跳级：不能在没有 `### 3.2` 的情况下直接出现 `#### 3.2.1`
- 二、三级标题编号的章号必须与当前章一致

### 2.3 标题之间必须有说明文字

二级、三级标题后，直到下一个标题出现之前，至少应有一段正文描述。

不建议以下写法：

```md
### 3.2 RSS 源管理
#### 3.2.1 RSS 源注册与配置
```

建议至少补一句过渡说明，例如：

```md
### 3.2 RSS 源管理
本节围绕 RSS 源的注册、抓取控制与状态监控展开。
#### 3.2.1 RSS 源注册与配置
```

## 3. 摘要与关键词

### 3.1 中文摘要

- `## 摘要` 下先写摘要正文
- 摘要正文结束后使用 `<br>` 换行，再写关键词
- 关键词行建议写为：`**关键词**  词1  词2  词3`

示例：

```md
## 摘要

这里是摘要正文最后一段。<br>
**关键词**  智能情报系统  RSS 聚合  大语言模型
```

### 3.2 英文摘要

- `## ABSTRACT` 下先写英文摘要正文
- 英文摘要正文结束后使用 `<br>` 换行，再写关键词
- 关键词行建议写为：`**KEY WORDS**  ...`

## 4. 图片与题注

### 4.1 基本规则

- 每张图片都必须有题注
- 图片题注通常放在图片下方
- 题注格式建议为：`图 3-1 系统参与者图`
- 题注最后一个字应为“图”
- 图片既可以是相对 `thesis.md` 可解析的本地文件，也可以是可直接访问的 `http/https` URL
- 若使用 URL 图片，导出时会联网抓取；建议优先使用稳定的 PNG/JPG/BMP/GIF 直链

### 4.2 推荐写法

```md
![系统参与者图](image-1.png)

图 3-1 系统参与者图
```

也支持 URL 直链，例如：

```md
![系统参与者图](https://example.com/system-actor.png)

图 3-1 系统参与者图
```

### 4.3 表格单元格内嵌图

若图片出现在 Markdown 表格单元格内：

- 仍然必须保留图片题注
- 推荐在同一单元格中使用 `<br>` 分隔图片和题注
- 题注依然使用 `图 X-Y ...图` 形式

示例：

```md
| 实际结果 | 如图所示<br>![图 6-1 单RSS源管理页图](test/单rss源.png)<br>图 6-1 单RSS源管理页图 |
```

## 5. 表格与题注

### 5.1 基本规则

- 每个表格都必须有题注
- 表格题注必须放在表格上方
- 题注格式建议为：`表 6-1 新增RSS测试用例表`
- 题注最后一个字应为“表”

### 5.2 推荐写法

```md
表 6-1 新增RSS测试用例表

| 字段 | 说明 |
| --- | --- |
| ... | ... |
```

## 6. 自动检查覆盖范围

以下规则应由检查工具自动覆盖：

1. 标题层级提取与完整目录输出
2. 一级/二级/三级标题格式与编号合法性
3. 二、三级标题后必须先有正文描述
4. 摘要 / ABSTRACT 的 `<br> + 关键词` 规则
5. 普通图片必须有下方题注；本地图片需存在，URL 图片需为可访问的 `http/https` 直链
6. Mermaid / PlantUML 等图示代码块会按图片处理，因此也必须有下方题注
7. 普通表格必须有上方题注
8. 表格单元格内图片必须可解析（本地文件存在或 URL 合法），且应有同单元格题注
9. 图表编号唯一性与基本格式合法性

## 7. 图示代码块（Mermaid / PlantUML）

- 若代码块语言标记为 `mermaid`、`plantuml`、`puml` 或 `uml`，导出脚本会先提取代码块，再通过 API 渲染成 PNG 临时图片，最后以普通图片方式嵌入 DOCX
- 检查脚本会把这几类图示代码块按“图片对象”处理，因此同样要求其下方提供 `图 X-Y ...图` 形式的题注
- Mermaid 默认通过 API 直接渲染；PlantUML 也优先走 API，不依赖本地安装 PlantUML 运行链路
- PlantUML 在渲染前会统一追加以下前缀：

```text
skinparam backgroundColor white
skinparam defaultFontName "Times New Roman"
scale 2
```

- 导出完成后会自动删除本次渲染产生的中间图片与临时目录，不保留在论文工作区
- 若代码块不是上述语言标记，则仍按普通代码块处理

## 8. 当前导出链路的已知约束

- 图片统一使用 PNG/JPG 等位图资源，不继续走 SVG 导出链路
- Markdown 表格单元格中的图片会正常渲染，但列宽计算应忽略图片 Markdown 与尺寸，仅按文本内容分配
- 目录由导出脚本在 `## 目录` 位置自动生成，不直接手写目录项
- 自动目录从“第一章”开始，不收录“摘要”“ABSTRACT”“目录”
- Mermaid / PlantUML 当前默认通过 `https://kroki.io` 渲染为 PNG；若后续需要切换私有服务，可通过导出脚本参数 `--kroki-base` 覆盖

## 9. 建议写作习惯

- 图表前后留空行，避免和正文黏连
- 标题下先写一句引导语，再放图、表、公式或列表
- 公式、图、表、测试用例等对象尽量按章编号，便于后续统一核查
- 修改章节结构后，先跑检查工具，再执行 md2doc 导出
